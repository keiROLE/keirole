# 我如何用 LLM Wiki 思想重构个人知识库

> 知识不是"存"进去的，是"编译"出来的。

## 问题：一个 150+ 文件的死水仓库

我的 Obsidian 仓库曾是一个典型的"数字垃圾堆"——150 多篇 Markdown 文件平铺在根目录或浅层文件夹中，没有任何结构逻辑：

- **找不到**：十个根级文件夹（ai/、认知记录/、副业/、每周复盘/……）彼此之间没有分类关系，连我自己都记不清哪个文件装了什么。
- **连不上**：全仓库只有约 10 个文件使用了 Wiki-link，知识是孤岛，无法形成网络。
- **长不大**：没有概念索引、没有实体目录、没有对比矩阵。每次往里面塞新笔记，旧笔记就被埋得更深。
- **"杂"和"未命名"**：大量文件名含"杂"或"未命名"的文件，成了知识的坟墓——"先存起来，以后再说"。

这就是传统笔记方法的死穴：**它假设知识是静态的，可以像文件一样"存放"。**

## 灵感：Karpathy 的编译器隐喻

Andrej Karpathy 有一个经典观点：**知识管理应该像一个编译器，而不是一个文件夹。**

传统笔记是"存储"——你把东西放进去，希望以后能找到。LLM Wiki 是"编译"——你把原始素材丢进去，AI 从中提取知识单元，结构化、去重、建立交叉引用，生成可复用的知识条目。

两者的根本区别在于：

| 维度 | 存储思维 | 编译思维 |
|------|---------|---------|
| 输入 | 整理好的笔记 | 原始素材（剪藏、草稿、对话记录） |
| 过程 | 手动分类、打标签 | AI 提取、结构化、建立链接 |
| 输出 | 文件系统中的文件 | 带 frontmatter 的标准化知识卡片 |
| 增长 | 线性增加 | 网络效应，条目间互相引用产生复利 |

## 架构：三层系统

我设计了一个三层架构，每个层级有明确的职责：

### Layer 0：0_Inbox/ — 原始素材层

所有未经处理的输入统一汇入这里：网页剪藏、PDF 导出、语音转文字、AI 对话记录、任何"先存起来再说"的东西。

**核心规则：0_Inbox/ 里的文件不允许编辑。** 它们是事实来源，保持原始状态意味着永远不会丢失信息。

### Layer 1：1_Wiki/ — 知识编译层

AI 维护的核心知识库。内容由编译产生，不是手动书写。

| 子目录 | 内容 | 回答的问题 |
|--------|------|-----------|
| concepts/ | 概念卡片 | 这是什么？ |
| entities/ | 实体卡片 | 哪个东西/人/组织？ |
| comparisons/ | 对比卡片 | A 和 B 有啥区别？ |
| frameworks/ | 方法论卡片 | 怎么做？ |
| resources/ | 资源卡片 | 有哪些工具/网站？ |
| templates/ | 5 种条目模板 | 新建条目时套用 |
| archive/ | 已处理/废弃 | 历史版本 |

目前编译了 **76 个条目**（29 个概念 + 3 个实体 + 5 个对比 + 30 个框架 + 9 个资源），覆盖了校园团队运营、自媒体全栈、认知体系、副业矩阵、AI 工具等多个领域。

### Layer 2：2_Schema/ — 配置层

知识库的"操作系统"。包含全局规则、命名约定、AI 工作流、用户画像等。所有 AI 任务和人类操作都必须遵循本层规范。

## 编译流水线

我构建了四个 Skill 来实现完整的编译管线，全部由用户主动触发：

### `/by-1` 编译单篇

读取用户指定的单篇文档，提取知识单元，编译到 1_Wiki/ 对应条目，更新 index.md。

```
用户："编译这篇" → AI 读取 → 分类 → 查重 → 编译 → 更新索引 → 输出报告
```

### `/by-a` 编译全部

自动扫描全仓库找未编译文件，按主题分组并行批量编译。

```
用户："编译全部" → AI 扫描 → 分组 → 并行编译 → 统一更新索引 → 输出报告
```

### `/jkjc-1` 健康检查

全面扫描 1_Wiki/，检测 broken links、hallucination、内容遗漏、结构问题，以及**知识图谱补链**（发现孤立节点并推荐桥接链接）。

### `/jh-1` 主题聚合

给定主题，扫描所有条目相关内容，生成主题知识地图，建议合并或新增条目。

## 编译规则的精妙之处

### 1. 编译溯源

每个条目都有 `source` 字段，标明从哪篇原始笔记编译而来。这意味着：

- 你可以回溯知识的来源
- 如果编译质量有问题，可以找到原文重新审视
- 永远不会丢失原始上下文

### 2. 置信度自评

每个条目标注 `confidence: high/medium/low`，让 AI 对自己的编译质量诚实。这避免了"AI 什么都敢说"的问题。

### 3. 强制交叉引用

frontmatter 中的 `related` 字段强制新条目与已有条目建立联系。这不是事后整理，而是**在创建时就要求网络化的思维**。

### 4. 原文不动原则

所有原文件夹（ai/、认知记录/、副业/……）**全部原位保留，不删除、不重命名、不移动**。AI 编译时跨目录扫描所有 .md 文件作为知识来源，读取后写入 1_Wiki/，原文件不变。

这个设计消除了迁移风险——即使编译出错，原始数据毫发无损。

## 知识复利的数学

传统笔记的增长是线性的：每篇新笔记 = +1 个文件。

LLM Wiki 的增长是指数级的：每篇新笔记 = +N 个知识单元 + M 条交叉引用。

当条目数达到一定规模时，**链接的价值超过了单个条目的价值**。你不需要重新阅读 150 篇原始笔记——你只需要在 index.md 的导航表中找到对应条目，它已经是你需要的全部精华，并且附带了与其他条目的关联。

## 经验教训

### 做了什么对的

- **不破坏原文**：原文件夹全部保留，编译是叠加而非迁移
- **结构化模板**：5 种条目类型各有模板，保证了编译一致性
- **用户主动触发**：所有编译操作必须由用户发起，AI 不自动执行
- **产出隔离**：AI 的分析报告和修改建议统一存放在 `cc产出/`，不直接修改 wiki 条目
- **元知识沉淀**：`沉淀_Obsidian.md` 记录了可复利方法和踩坑日志，系统在学习自身

### 遇到的坑

- **Windows 中文路径乱码**：bash 的 `ls`/`glob` 在中文路径下偶尔失效，需要用 `cmd /c "dir"` 替代
- **空骨架文件**：某些文件从子文件夹移动到根目录后只剩标题，移动前必须用 `wc -l` 确认文件大小
- **跨文件重复**：合并重组时发现同一概念在多个文件中重复出现，需要在编译阶段就识别去重
- **Skill 工具加载后未执行**：Workflow 并行分发 agent 时，单个 agent 的工具加载可能超时，需要重试机制

## 总结

LLM Wiki 不是一种新的笔记方法，而是一种新的**知识生产方式**。

它把人的角色从"整理者"提升为"决策者"——你不再需要纠结文件名怎么取、文件夹怎么建、标签怎么打。你只需要提供原始素材，说一声"编译"，AI 负责提取、结构化、链接。

而你需要做的是：阅读编译结果，判断质量，决定下一步方向。

**知识不是存出来的，是编译出来的。** 这句话听起来像口号，但它改变了你与知识的关系——从被动收藏，到主动提炼。
