# fp-1 与 fp-2：我的 AI 迭代升级思想

最近我在 Claude Code 里创建了两个 skill：`/fp-1` 和 `/fp-2`。它们表面上都是"复盘"，但分工完全不同——一个复盘**事**，一个复盘**人**。

这个设计不是拍脑袋来的，而是经历了几个迭代阶段逐步演化出来的。

## 第一阶段：只复盘事

最开始只有一个 `/fp-1`。它的目标是：每次和 AI 协作结束后，自动提取可复利的方法论和踩过的坑，写入项目沉淀和全局沉淀。

这解决了一个真实痛点：和 AI 聊了一小时解决了问题，但下次遇到类似场景还得重新摸索。知识存不住，等于白干。

`/fp-1` 的核心逻辑是**分层沉淀**：
- 项目级沉淀：针对当前项目的具体经验和教训
- 全局沉淀：跨项目通用的方法论（如 PowerShell BOM 编码、Bash heredoc 反引号冲突）

## 第二阶段：发现"事"的背后还有"人"

用了一段时间 `/fp-1` 后，我发现一个问题：有些坑反复踩，不是因为方法论没沉淀下来，而是因为**沟通方式有问题**。

比如：
- 指令模糊 → AI 理解偏差 → 返工 → 踩坑 → 记录到 fp-1
- 信息缺失 → AI 反复追问 → 浪费时间 → 还是那个坑

换句话说，**很多"事"的问题，根源在"人"的沟通上**。只复盘事，治标不治本。

于是有了 `/fp-2`。

## 第三阶段：双环学习

`/fp-1` 和 `/fp-2` 构成了一个**双环学习系统**：

| | fp-1（复盘事） | fp-2（复盘人） |
|---|---|---|
| **对象** | 项目经验、技术坑、方法论 | 用户的沟通方式、指令质量 |
| **产出** | 沉淀文档（可复利方法 + 踩坑日志） | 诊断报告 + 记仇本（长期习惯追踪） |
| **层级** | 战术层：这次怎么做的 | 战略层：以后怎么说更好 |
| **频率** | 每次协作后 | 每次对话后 |

双环学习的概念来自组织理论：
- **单环学习**：发现错误 → 修正行动（fp-1 做的事）
- **双环学习**：发现错误 → 反思背后的假设和规则（fp-2 做的事）

## 迭代心法

这套设计背后有几个迭代原则：

### 1. 从痛点出发，不是从概念出发

不是先想"我应该做一个复盘系统"，而是先感受到"每次协作完知识就丢了"这个痛。痛点越真实，设计越精准。

### 2. 最小可用，逐步演化

先做 `/fp-1` 跑通"复盘-沉淀"这个闭环，发现缺口后再补 `/fp-2`。不是一次性设计完美系统。

### 3. 分层解耦

fp-1 管项目沉淀，fp-2 管沟通习惯，两者互不干扰但互为补充。解耦之后各自可以独立演进。

### 4. 可执行优于可记录

记录本身没有价值，价值在于**下次能少踩一个坑**。所以 fp-1 强调"可复利方法"而不是"发生了什么"，fp-2 强调"具体改写示例"而不是"你应该说清楚"。

## 写在最后

这两个 skill 的本质，是把**人机协作**当成一个可以持续优化的系统来经营。

不是"我用 AI 干活"，而是"我和 AI 一起成长"。

fp-1 让经验留得下来，fp-2 让沟通变得更好。两者叠加，下一次协作就会比这一次更高效。

这就是我的 AI 迭代升级思想：**用复盘驱动进化，用进化减少复盘的必要性。**
